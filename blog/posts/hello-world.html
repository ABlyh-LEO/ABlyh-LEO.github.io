Getting Started with Embedded Systems — My RoboMaster Journey

2025-01-15 Embedded · STM32

When I first joined the RoboMaster team at USTC, I had no idea how much I would learn in just one year. This post is a reflection on my journey from a beginner to leading the embedded development of the sentinel robot.

Why Embedded Systems?

Coming from a mathematics-oriented background (Information & Computational Science), my initial exposure to programming was mostly algorithmic — C++ for ICPC contests, data structures, and problem solving. Embedded systems felt like a completely different world: hardware, real-time constraints, communication protocols, and physical robots that could actually move.

The transition wasn't easy. Reading vendor datasheets is nothing like reading algorithm tutorials. But the first time I saw a motor spin because of code I wrote, I was hooked.

The Sentinel Robot

The sentinel robot in RoboMaster is a fully autonomous defense unit. It sits on a rail, scans for enemy robots, and automatically engages targets. My responsibilities included:

  • Chassis control: Designing control algorithms for a special-configuration chassis that moves along a rail system
  • Gimbal stabilization: Keeping the gimbal steady while the chassis moves, using PID control and sensor fusion
  • Communication: Implementing CAN FD, UART, and USB virtual serial port communication between modules
  • System integration: Coordinating with the vision team and making sure all subsystems work together

Key Lessons Learned

1. Read the Datasheet

There is no substitute for reading the manufacturer's documentation. The STM32H7 reference manual is over 3,000 pages long. You don't need to read all of it, but knowing where to find the right information is a skill in itself.

2. C++ in Embedded is Different

C++ on a microcontroller is not the same as C++ on a PC. Memory is limited, dynamic allocation is risky, and you need to think about interrupt safety. Templates and classes are useful, but you must understand what the compiler generates.

3. Communication is Everything

A robot is a distributed system. The chassis controller, gimbal controller, vision module, and referee system all need to talk to each other. Getting the communication protocols right — timing, error handling, bandwidth — is critical for reliability.

4. Debugging Hardware is Hard

When your code crashes on a PC, you get a stack trace. When it crashes on a microcontroller, the robot might just stop moving — or worse, start moving in the wrong direction. Logic analyzers, oscilloscopes, and serial logging become your best friends.

Looking Forward

The RoboMaster experience taught me far more than just embedded programming. It taught me how to work in a multidisciplinary team, how to manage a complex project over months, and how to stay calm when things break right before a competition. These are skills I will carry with me wherever I go next.

If you're thinking about getting into embedded systems, my advice is simple: find a project, get a development board, and start building. The STM32 ecosystem has excellent documentation and a supportive community. You'll be surprised by how much you can learn in a short time.
